Name

Generic — Acts as either a message source or a message sink

Usage

Components can be placed at any point along a route. A route must start with the Generic and typically finishes with the Generic.

Properties

Table 6 describes the properties you can specify using the properties editor.

Table 6. Generic Properties

NameDescription
UriSpecifies the URI for the Generic. The URI determines the type of component used to read or send message data. For more information see Components Overview.
Id

Specifies a unique identifier for the Generic.

The tooling automatically generates an id for a node when it is created, but you can remove that id or replace it with your own. The Camel debugger requires all nodes with a breakpoint set to have a unique id.

You can use the id to refer to components in your Camel XML file.

Exchange Pattern

Specifies the pattern of the exchange:

  • InOnly—Standard one-way exchange. Consumer sends message exchange to service provider, and service provider responds with status.

  • RobustInOnly—Reliable one-way exchange. Consumer sends message exchange to service provider, and service provider responds. If response is a status message, the exchange is completed. If the response is a fault message, the consumer must respond with a status message.

  • InOut—Equivalent to a request-reply exchange. Consumer sends the initial message to the service provider, and the service provider responds with either a response message or a fault message. The consumer then responds with a status message.

  • InOptionalOut—Standard two-way exchange in which the service provider's response is optional.

  • OutOnly—Reverse of InOnly in that the service provider sends out a message, which cannot trigger a fault message. This pattern supports event notification.

  • RobustOutOnly—Similar to OutOnly, except that it can trigger a fault message. The outbound message initiates the exchange.

  • OutIn—Reverse of InOut in that the service provider transmits the request initiating the exchange.

  • OutOptionalIn—Reverse of InOptionalOut in that the service provider initiates an exchange by sending an outbound message, and consumer response is optional.

DescriptionSpecifies a text description for the node. This description is included in the generated XML file, but it is informational only. It is not used by Apache Camel.
